# ドキュメント管理ルール

## 目的

チームでのドキュメント管理を効率化し、情報の一元化・最新化を維持する。

## 基本方針

1. **構造化**: すべてのドキュメントはMarkdown形式で管理
2. **一元化**: このリポジトリを唯一の情報源（Single Source of Truth）とする
3. **継続改善**: 定期的にレビュー・更新する

## フォルダ構成

| フォルダ | 内容 | 更新頻度 |
|---------|------|----------|
| `00_policy/` | 運用ルール | 低（安定後は月1程度） |
| `01_discovery/` | 発見・検証フェーズ | 高（検証中は毎日） |
| `02_strategy/` | 戦略フェーズ | 中（週1程度） |
| `03_development/` | 開発フェーズ | 高（開発中は毎日） |
| `04_growth/` | 成長フェーズ | 中（週1程度） |
| `99_archive/` | アーカイブ | 随時 |

## 命名規則

### ファイル名
- 小文字英数字とハイフンを使用: `target-persona.md`
- 日付を含む場合: `2025-01-15-meeting-note.md`
- 日本語ファイル名は使用しない

### ドキュメント構造
```markdown
---
title: ドキュメントタイトル
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: draft | review | approved
---

# タイトル

## セクション1

内容...
```

## ステータス定義

| ステータス | 意味 | 次のアクション |
|-----------|------|----------------|
| `draft` | 作成中・未完成 | 内容を充実させる |
| `review` | レビュー待ち | レビュー後に修正 |
| `approved` | 承認済み・確定 | 必要に応じて更新 |

## コミットメッセージ規則

```
[カテゴリ] 変更内容

例:
[docs] ターゲットペルソナを追加
[hypothesis] 課題仮説H2を検証済みに更新
[fix] typo修正
[chore] フォルダ整理
```

### カテゴリ一覧
- `docs`: ドキュメント追加・更新
- `hypothesis`: 仮説の更新
- `strategy`: 戦略関連の更新
- `dev`: 開発関連
- `fix`: 修正
- `chore`: 雑務・整理

## レビュープロセス

### 変更の種類別フロー

| 変更種類 | フロー |
|----------|--------|
| 軽微な修正（typo等） | 直接mainブランチにコミット |
| 通常の更新 | ブランチ作成 → コミット → mainにマージ |
| 重要な変更 | Issue作成 → ブランチ → PR → レビュー → マージ |

### 重要な変更の例
- 仮説の大幅な変更・ピボット
- ターゲットの変更
- ビジネスモデルの変更
- ロードマップの大幅な変更

## 古いドキュメントの扱い

1. 内容が古くなったドキュメントは `99_archive/` に移動
2. 移動時にファイル名に日付を追加: `2025-01-archived-old-hypothesis.md`
3. 元の場所から参照リンクがあれば更新

## 定期レビュー

### 週次
- [ ] 仮説の進捗確認
- [ ] 実験・検証結果の記録
- [ ] 学びの追記

### 月次
- [ ] 全ドキュメントの棚卸し
- [ ] 古い情報のアーカイブ
- [ ] 運用ルールの見直し
