# AI活用ガイド

## 概要

このテンプレートは **AIツールとの対話を通じてドキュメントを作成・改善していく** ことを前提に設計されています。

曖昧なアイデアや仮説をAIに伝え、対話を通じて構造化されたドキュメントに落とし込んでいきます。また、メモや議事録をAIで整理し、ドキュメントを継続的に改善していきます。

## 使用するAIツール

| ツール | 主な用途 | 特徴 |
|--------|----------|------|
| **Cursor** | コード開発、ドキュメント編集 | エディタ統合、リポジトリ全体を理解 |
| **Claude Code** | ドキュメント作成・編集、コードレビュー | 長文対応、分析力が高い |
| **GitHub Copilot** | コード補完、ドキュメント作成支援 | VS Code/JetBrains統合 |
| **OpenAI Codex** | コード生成、ドキュメント作成 | API連携 |
| **ChatGPT** | アイデア出し、壁打ち | 汎用的、手軽 |
| **Perplexity** | リサーチ、競合調査 | Web検索統合 |

## AIとの対話ワークフロー

### 基本の流れ

```
① 曖昧なアイデア・仮説を持っている
    ↓
② AIに伝えて対話しながら整理・構造化
    ↓
③ ドキュメントとして保存
    ↓
④ メモ・議事録をAIで整理してドキュメントを改善
    ↓
⑤ 繰り返し（継続的に洗練）
```

### ワークフロー1: アイデアからドキュメントを作成

**Step 1**: 頭の中にあることをそのままAIに伝える

```
あなた:
「30代のエンジニアで副業したい人向けのサービスを考えてる。
AIがその人に合ったプロジェクトをレコメンドしてくれる感じ。
まだ曖昧なんだけど、仮説として整理してほしい」
```

**Step 2**: AIと対話して深掘り

```
AI:
「いくつか確認させてください：
1. 副業したい理由は収入？スキルアップ？
2. どんなプロジェクト（受託？OSS？）を想定？
3. なぜ既存のサービスではダメ？」

あなた:
「収入が主。受託案件を想定してる。
既存サービスは案件多すぎて選べないのが課題」
```

**Step 3**: ドキュメント形式で出力

```
AI:
「整理すると以下のようになります。hypothesis.md に追記しますか？

### H1: ターゲット仮説
| **仮説** | 30代エンジニアで副収入を得たい人がターゲット |
| **根拠** | ... |
| **状態** | 🔴 未検証 |

### H2: 課題仮説
| **仮説** | 案件が多すぎて自分に合うものを選べない |
...」
```

### ワークフロー2: メモ・議事録からドキュメントを改善

**Step 1**: ミーティングや調査のメモを `notes/` に保存

```
notes/meetings/2025-01-15-user-interview-tanaka.md
```

**Step 2**: AIにメモを読ませてドキュメントを更新

```
あなた:
「notes/meetings/ にある最近のユーザーインタビュー議事録を読んで、
hypothesis.md の検証結果を更新して。
新しい発見があれば learnings.md にも追記して」
```

**Step 3**: AIが複数ドキュメントを横断的に更新

```
AI:
「3件の議事録を確認しました。以下を更新します：

1. hypothesis.md
   - H1（ターゲット仮説）: 🟢 検証済みに更新
   - H2（課題仮説）: 新しい課題を1つ追加

2. learnings.md
   - 学び: 「想定より40代のニーズが高い」を追記

3. target-persona.md
   - 年齢層を「30-40代」に修正

実行しますか？」
```

### ワークフロー3: 壁打ち・ブレインストーミング

```
あなた:
「今の仮説に自信がない。hypothesis.md を読んで、
穴がないか、見落としている観点がないか指摘して」

AI:
「hypothesis.md を確認しました。以下の点が気になります：

1. 競合との差別化が弱い
   - 「AIレコメンド」だけでは既存サービスと同じでは？

2. 価格仮説が未検証
   - ターゲットはいくらまで払う？

3. 「選べない」の定義
   - 時間がない？選び方がわからない？

これらを深掘りしますか？」
```

## AIとの連携方法

### 1. CLAUDE.md を活用

リポジトリルートの `CLAUDE.md` にはプロジェクトの概要とルールが記載されています。AIツールはこのファイルを参照してコンテキストを理解します。

### 2. プロンプト集を活用

`prompts/` フォルダに用途別のプロンプトがあります：

```
prompts/
├── ideation/     # アイデア発想
├── analysis/     # 分析
└── development/  # 開発
```

### 3. コンテキストを提供

AIに依頼する際は、関連ドキュメントを読み込ませると精度が上がります：

```
以下のドキュメントを踏まえて、XXXしてください：
- docs/01_discovery/hypothesis.md
- docs/01_discovery/target-persona.md
```

### 4. メモ・議事録を活用

`notes/` フォルダのメモをAIに読ませて、ドキュメントを改善させます：

```
notes/meetings/2025-01-15-kickoff.md を読んで、
関連するドキュメントを更新して
```

## フェーズ別AI活用

### Discovery（発見）フェーズ
- アイデアのブレインストーミング
- 仮説の言語化・構造化
- 競合リサーチの要約
- ペルソナの具体化

### Strategy（戦略）フェーズ
- バリュープロポジションの整理
- ビジネスモデルの検討
- MVP機能の優先順位付け
- ロードマップ作成

### Development（開発）フェーズ
- 要件定義書の作成
- 設計ドキュメント作成
- コード生成・レビュー
- テストケース作成

### Growth（成長）フェーズ
- データ分析・インサイト抽出
- 実験結果の解釈
- 改善案の提案
- レポート作成

## AIに依頼する際のコツ

### 良い依頼の例

```
【コンテキスト】
当プロジェクトはXXXをターゲットにした△△サービスです。
現在、以下の仮説を検証中です：（hypothesis.mdの内容）

【依頼】
上記を踏まえ、ユーザーインタビューで確認すべき質問を10個提案してください。

【出力形式】
- 質問文
- その質問で確認したい仮説
```

### 悪い依頼の例

```
インタビュー質問を考えて
```
（コンテキストがないため、汎用的すぎる回答になる）

## 注意事項

### やるべきこと
- AIの出力は必ず人間がレビューする
- 重要な意思決定はAI任せにしない
- 出典が必要な情報はファクトチェックする
- プロンプトの改善履歴を記録する

### やってはいけないこと
- 機密情報・個人情報をAIに入力しない
- AIの出力をそのまま確定情報として扱わない
- 著作権のある長文をそのまま入力しない

## プロンプトの管理

### 新しいプロンプトを追加する

1. `prompts/` の適切なフォルダに追加
2. 以下のフォーマットで作成：

```markdown
---
title: プロンプト名
purpose: 目的
input: 必要な入力
output: 期待する出力
---

# プロンプト名

## 使用方法
（いつ、どのように使うか）

## プロンプト本文
（実際のプロンプト）

## 使用例
（入出力の例）
```

3. `prompts/README.md` の一覧に追加

### プロンプトの改善

プロンプトを改善したら、ファイル内の「改善履歴」に記録：

```markdown
## 改善履歴

| 日付 | 変更内容 | 理由 |
|------|----------|------|
| 2025-01-15 | 出力形式を詳細化 | 曖昧な回答が多かったため |
```
