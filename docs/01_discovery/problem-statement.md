# 課題定義

## 概要

顧客が抱えている課題を明確に定義します。課題の解像度が高いほど、解決策の精度も上がります。

---

## 課題ステートメント

### 一言で表す課題

```
【ターゲット】は【状況】において、【課題】に困っている。
なぜなら【根本原因】だからである。
```

<!--

【新規事業担当者】は【アイデアを形にしたい時】において、
【何から始めればいいかわからない】に困っている。
なぜなら【MVP開発の経験がなく、効率的な進め方を知らない】からである。

## 課題の優先順位

複数の課題がある場合、どれから解決するか優先順位をつけます。

| 順位 | 課題 | 深刻度 | 頻度 | 解決可能性 | 優先度 |
|------|------|--------|------|------------|--------|
| 1 | | 高/中/低 | 高/中/低 | 高/中/低 | ⭐⭐⭐ |
| 2 | | | | | ⭐⭐ |
| 3 | | | | | ⭐ |

### 評価基準

- **深刻度**: 課題が顧客に与える影響の大きさ
- **頻度**: 課題が発生する頻度
- **解決可能性**: 自社で解決できる可能性

## Jobs to be Done (JTBD)

顧客が「雇いたい」と思う仕事（ジョブ）を整理します。

### 機能的ジョブ
> 【状況】の時、【目的】のために【行動】したい

<!--
記入例:

> 【感情】を感じたい / 感じたくない

## 参考: 課題発見のフレームワーク

### 5 Whys（なぜを5回）
表面的な課題から根本原因を探る

```
課題: 
→ Why? 
→ Why? 
→ Why? 
→ Why? 
→ Why? （根本原因）
```

### 課題の4象限
```
                高い影響
                   |
    緊急で        |       重要だが
    重要          |       緊急でない
                   |
 ─────────────────┼─────────────────
                   |
    緊急だが      |       重要でも
    重要でない    |       緊急でもない
                   |
                低い影響
```
